# Feature Specification: TODO App

**Feature Branch**: `001-todo-app`
**Created**: 2026-03-15
**Status**: Draft
**Input**: User description: "簡単なTODOアプリを作成したい。個人のタスク管理用。CRUD操作、完了/未完了の切り替え、フィルタリング機能を持つ。認証不要の単一ユーザーアプリ。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - TODO の基本操作 (Priority: P1)

ユーザーとして、TODO を作成・表示・編集・削除できる。これにより日々のタスクを一元管理し、やるべきことを漏れなく把握できる。

**Why this priority**: TODO アプリの根幹機能であり、これがなければアプリとして成立しない。この機能だけで最小限のタスク管理が可能になる。

**Independent Test**: TODO の作成・表示・編集・削除を一通り操作し、データが正しく永続化されていることを確認することで独立してテストできる。

**Acceptance Scenarios**:

1. **Given** TODO が存在しない状態, **When** ユーザーがタイトルを入力して追加ボタンを押す, **Then** 新しい TODO が一覧に表示される
2. **Given** TODO が 1 件以上存在する状態, **When** ユーザーが一覧画面を開く, **Then** 全ての TODO がタイトルとステータス付きで表示される
3. **Given** TODO が存在する状態, **When** ユーザーが一覧上の TODO タイトルをクリックする, **Then** その場でインライン編集モードになり、変更を保存すると一覧に反映される
4. **Given** TODO が存在する状態, **When** ユーザーが TODO を削除する, **Then** 確認ダイアログが表示され、確認後に一覧から該当 TODO が消える
5. **Given** ユーザーがタイトル未入力で追加を試みる, **When** 追加ボタンを押す, **Then** エラーが表示され TODO は追加されない

---

### User Story 2 - 完了管理とフィルタリング (Priority: P2)

ユーザーとして、TODO を完了/未完了に切り替え、ステータスでフィルタリングできる。これにより残タスクの把握と進捗管理が容易になる。

**Why this priority**: 基本的な CRUD に次いで重要な機能。タスクの進捗を管理することで、アプリの実用性が大幅に向上する。

**Independent Test**: TODO を複数作成し、一部を完了に切り替えた後、フィルタリングで「未完了のみ」「完了のみ」「全件」を切り替えて正しく絞り込めることを確認する。

**Acceptance Scenarios**:

1. **Given** 未完了の TODO が存在する状態, **When** ユーザーがチェックボックスをクリックする, **Then** TODO が完了状態になり視覚的に区別される
2. **Given** 完了済みの TODO が存在する状態, **When** ユーザーがチェックボックスをクリックする, **Then** TODO が未完了状態に戻る
3. **Given** 完了・未完了の TODO が混在する状態, **When** ユーザーが「未完了」フィルタを選択する, **Then** 未完了の TODO のみが表示される
4. **Given** フィルタが適用されている状態, **When** ユーザーが「全件」フィルタを選択する, **Then** 全ての TODO が表示される

---

### Edge Cases

- 空文字やスペースのみのタイトルを入力した場合、バリデーションエラーを表示する
- 非常に長いタイトル（200 文字超）を入力した場合、入力を制限する
- TODO が 0 件の状態で一覧画面を表示した場合、空状態のメッセージを表示する
- フィルタ適用中に新しい TODO を追加した場合、フィルタ条件に応じて表示が正しく反映される
- ページリロード後も TODO のデータとステータスが保持される

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: ユーザーはタイトルを入力して新しい TODO を作成できること
- **FR-002**: システムは全ての TODO を一覧表示できること
- **FR-003**: ユーザーは一覧上で TODO のタイトルをクリックしてインライン編集できること
- **FR-004**: ユーザーは TODO を削除できること（確認ダイアログで確認後に実行）
- **FR-005**: ユーザーは TODO の完了/未完了をトグル切り替えできること
- **FR-006**: ユーザーはステータス（全件/完了/未完了）で TODO をフィルタリングできること
- **FR-007**: システムは TODO データを永続的に保存し、再アクセス時にも保持すること
- **FR-008**: システムは空文字のタイトルでの TODO 作成を拒否すること
- **FR-009**: 完了済みの TODO は視覚的に区別して表示すること

### Key Entities

- **TODO**: ユーザーが管理するタスクの単位。タイトル（必須・文字列・最大200文字）、完了ステータス（真偽値）、作成日時、更新日時を持つ

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーが TODO の作成から一覧確認までを 5 秒以内に完了できる
- **SC-002**: 完了/未完了の切り替え操作が 1 アクション（ワンクリック）で完了する
- **SC-003**: フィルタリング切り替え後、結果が即座に（体感 1 秒以内）反映される
- **SC-004**: 100 件の TODO が存在する状態で、一覧表示が 2 秒以内に完了する
- **SC-005**: ページリロード後も全ての TODO データとステータスが 100% 保持される

## Clarifications

### Session 2026-03-15

- Q: TODO 削除時の確認動作は？ → A: 確認ダイアログを表示してから削除する
- Q: TODO の編集操作の方法は？ → A: 一覧上でタイトルをクリックしてインライン編集する

## Assumptions

- 単一ユーザーのアプリケーションであり、認証・認可は不要
- TODO のタイトル最大長は 200 文字とする
- 並び替え機能は本スコープに含まない（作成日時の降順で固定表示）
- カテゴリ・タグ・期限などの追加属性は本スコープに含まない
- オフライン対応は本スコープに含まない
